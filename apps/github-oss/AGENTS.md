## Learned User Preferences

- Prefer new projects persisted as JSON files under `data/projects/` (same idea as bundled seed projects) when the local server is used, instead of treating the browser store as the only source of truth for full repo lists.
- Discovery keywords added in the UI should stay multi-word when the user types phrases: normalize to lowercase and collapse repeated spaces, but do not replace spaces with hyphens so chips line up with GitHub phrase search in `github-discovery.js`.

## Learned Workspace Facts

- Serve the app with `npm run dev` or `npm start` from the repo (default `http://localhost:3847/`); `file://` is unreliable for root-relative scripts such as `github-discovery.js` and for `/api/*` persistence and listing.
- With the server available, full project payloads including `repos` are stored under `data/projects/<num>_<slug>.json`; the browser VFS (`localStorage` key `oss-vfs`) holds card metadata and backs the home list when the projects API is missing or returns an empty list for bootstrap.
- GitHub discovery is shared via `github-discovery.js` from both `index.html` and `project.html`; scope radios (metadata vs README-verify vs code), filters, and merge or dedupe behavior are intended to match between new-project search and add-repositories flows. Multi-word keywords are sent as quoted phrase terms in the GitHub query unless the chip already includes its own quoting.
- **Paired UI:** `index.html` “New Project Search” (`#searchModal`, id prefix `search`) and `project.html` “Add repositories” (`#extendSearchModal`, prefix `extend`) are intentionally mirrored. If you change discovery UI, defaults, or options wiring in one, update the other unless the task explicitly says otherwise.
- README is not fetched during general metadata search enrichment. In **README** scope mode (README-only), discovery fetches each candidate README, keeps repos only when the keyword phrase appears in that file body, and stores `readme` plus `readme_fetched` on those rows. On the project page, README also loads when you open a row (README tab) if not already stored; after load, `readme` plus `readme_fetched` are written to VFS and `data/projects/*.json` via `/api/projects/save`. Stack tab does the same for `packages` and `stack_fetched`. Reload uses saved data when present so GitHub is not hit again for those tabs.
- Dependency extraction during search defaults off; packages may be filled when the Stack tab is opened so `@package:` style filtering can stay empty until that path has run for a given repo. If “extract packages on search” is on, non-empty `packages` implies `stack_fetched` for those repos.
- Project page Ask AI (`project.html`): OpenRouter uses `/api/models` plus `/api/ai/ask` for a completed answer; Cursor uses `/api/cursor-models` (defaults plus optional comma-separated `CURSOR_MODELS` in server `.env`) and `/api/ai/ask` with `aiProvider: "cursor"` to return a copyable system/user prompt pack for Cursor Composer instead of calling OpenRouter for the reply.
